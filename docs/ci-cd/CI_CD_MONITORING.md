# CI/CD Monitoring and Status Reporting

This document describes how to monitor and verify the GitHub Actions CI/CD pipeline for UI SyncUp, including workflow status badges, migration execution metrics, and alerting capabilities.

## Workflow Status Badges

### GitHub Actions Status Badges

Status badges provide at-a-glance visibility of workflow health in the repository README:

1. **Deploy Workflow Badge**:
   ```markdown
   [![Deploy](https://github.com/YOUR_USERNAME/YOUR_REPO/actions/workflows/deploy.yml/badge.svg)](https://github.com/YOUR_USERNAME/YOUR_REPO/actions/workflows/deploy.yml)
   ```
   - Shows status of database migrations and deployments
   - Updates automatically on each workflow run
   - Click to view detailed workflow history

2. **CI Quality Checks Badge**:
   ```markdown
   [![CI Quality Checks](https://github.com/YOUR_USERNAME/YOUR_REPO/actions/workflows/ci.yml/badge.svg)](https://github.com/YOUR_USERNAME/YOUR_REPO/actions/workflows/ci.yml)
   ```
   - Shows status of TypeScript, ESLint, tests, and build
   - Updates automatically on each workflow run
   - Click to view detailed workflow history

3. **Badge Configuration**:
   - Replace `YOUR_USERNAME` with your GitHub username or organization
   - Replace `YOUR_REPO` with your repository name
   - Badges are automatically generated by GitHub Actions
   - No additional configuration required

4. **Badge Status Indicators**:
   - 🟢 **Passing**: All checks successful
   - 🔴 **Failing**: One or more checks failed
   - 🟡 **In Progress**: Workflow currently running
   - ⚪ **No Status**: No recent workflow runs

5. **Branch-Specific Badges** (Optional):
   ```markdown
   [![Deploy](https://github.com/YOUR_USERNAME/YOUR_REPO/actions/workflows/deploy.yml/badge.svg?branch=main)](https://github.com/YOUR_USERNAME/YOUR_REPO/actions/workflows/deploy.yml)
   ```
   - Add `?branch=main` to show status for specific branch
   - Useful for displaying production vs. preview status separately

### Viewing Badge Status

1. **In README**:
   - Badges appear at the top of the repository README
   - Click any badge to view detailed workflow history
   - Hover over badge to see tooltip with status

2. **In Pull Requests**:
   - Badges update to reflect PR branch status
   - Useful for quick PR health checks
   - Combined with PR status checks for comprehensive view

## Workflow Status Reporting

### Commit Status Checks

GitHub Actions automatically reports workflow status on commits and pull requests:

1. **Viewing Commit Statuses**:
   - Navigate to any commit in the repository
   - Scroll to the bottom to see "Checks" section
   - Each workflow job appears as a separate check
   - Status indicators: ✅ Success, ❌ Failure, 🟡 In Progress, ⚪ Pending

2. **Pull Request Status Checks**:
   - Open any pull request
   - Status checks appear in the "Checks" tab and at the bottom of the PR
   - Required checks must pass before merging (if branch protection is enabled)
   - Click on any check to view detailed logs

3. **Workflow Names and Jobs**:
   - **CI Quality Checks** workflow:
     - Job: `Quality Checks`
     - Steps: TypeScript, ESLint, Tests, Build
   - **Deploy** workflow:
     - Job: `Migrate Preview Database` (for non-main branches)
     - Job: `Migrate Production Database` (for main branch)

### Vercel Deployment Comments

Vercel automatically posts deployment status comments on pull requests:

1. **Preview Deployments**:
   - Vercel comments on PRs with preview deployment URLs
   - Comment includes: deployment status, preview URL, build logs link
   - Updates automatically as deployment progresses

2. **Production Deployments**:
   - Vercel posts production deployment status on main branch commits
   - Includes production URL and deployment details

3. **Verifying Vercel Integration**:
   ```bash
   # Check Vercel project settings
   # 1. Go to Vercel Dashboard → Project Settings
   # 2. Navigate to Git → GitHub Integration
   # 3. Verify:
   #    - Repository is connected
   #    - Production branch is set to "main"
   #    - Preview deployments are enabled for all branches
   #    - Comments on pull requests are enabled
   ```

### Workflow Logs

All workflow runs are logged and accessible:

1. **Accessing Logs**:
   - Navigate to repository → Actions tab
   - Select a workflow run from the list
   - Click on a job to view step-by-step logs
   - Expand any step to see detailed output

2. **Log Retention**:
   - GitHub retains workflow logs for 90 days
   - Download logs for offline analysis: Click "..." → "Download log archive"

3. **Migration Logs**:
   - Migration output is captured in workflow logs
   - Look for "Run database migrations" step
   - Logs include: migration files applied, SQL statements, errors (if any)

4. **Job Summaries**:
   - Workflow summaries appear at the top of each job
   - Migration jobs include: branch, commit, status, environment
   - Access via the "Summary" section of each workflow run

   - Verify average run times are within expected ranges

2. **Review Failed Workflows**:
   - Filter by "Failure" status
   - Review error logs

   - Optimize caching and dependencies if needed

2. **Review Deployment History**:
   - Check Vercel dashboard for deployment frequency
   - Review deployment success rate
   - Investigate any deployment failures

3. **Verify Environment Secrets**:
   - Ensure all required secrets are configured

1. **Review Workflow Configurations**:
   - Check for workflow file updates

   - Update documentation if needed

2. **Audit Access Controls**:
   - Review who has access to modify workflows

   - Update permissions if needed

## Troubleshooting Status Reporting Issues

### Commit Statuses Not Appearing

**Symptoms**: Workflow runs but status doesn't appear on commits/PRs

**Possible Causes**:
- Workflow file syntax error
- Workflow not triggered for the branch
- GitHub Actions disabled for repository

**Resolution**:
1. Check workflow file syntax: `gh workflow view ci.yml`
2. Verify workflow triggers match branch name
3. Check repository settings → Actions → General → Actions permissions

### Vercel Comments Not Appearing

**Symptoms**: Vercel deploys but doesn't comment on PRs

**Possible Causes**:
- Vercel GitHub integration not configured
- PR comments disabled in Vercel settings
- Vercel bot doesn't have repository access

**Resolution**:
1. Go to Vercel Dashboard → Project Settings → Git
2. Verify GitHub integration is connected
3. Enable "Comments on Pull Requests"
4. Verify Vercel bot has repository access

### Workflow Logs Not Accessible

**Symptoms**: Can't view or download workflow logs

**Possible Causes**:
- Insufficient repository permissions
- Logs expired (>90 days old)
- GitHub Actions quota exceeded

**Resolution**:
1. Verify you have "Read" access to Actions
2. Check log retention period
3. Review GitHub Actions usage quota

## Alerting and Notifications

### Alert Triggers

Set up alerts for critical migration events:

1. **Migration Failure Alerts**:
   - Trigger: `success: false` in metrics
   - Severity: Critical
   - Action: Immediate investigation required
   - Notification: Email, Slack, PagerDuty

2. **Performance Degradation Alerts**:
   - Trigger: `executionTimeMs > 30000` (30 seconds)
   - Severity: Warning
   - Action: Review migration complexity
   - Notification: Email, Slack

3. **High Failure Rate Alerts**:
   - Trigger: > 2 failures in 24 hours
   - Severity: High
   - Action: Review migration process
   - Notification: Email, Slack, team meeting

4. **Invalid Migration Files**:
   - Trigger: `invalidFiles > 0`
   - Severity: Medium
   - Action: Review and fix migration files
   - Notification: Email

### Alert Configuration Examples

1. **GitHub Actions Workflow Alert**:
   ```yaml
   - name: Alert on migration failure
     if: failure()
     run: |
       echo "::error::Migration failed - immediate attention required"
       echo "Branch: ${{ github.ref_name }}"
       echo "Commit: ${{ github.sha }}"
       echo "View logs: ${{ github.server_url }}/${{ github.repository }}/actions/runs/${{ github.run_id }}"
   ```

2. **Slack Alert with Metrics**:
   ```yaml
   - name: Send metrics to Slack
     if: always()
     uses: slackapi/slack-github-action@v1
     with:
       payload: |
         {
           "text": "Migration ${{ job.status }}: ${{ github.workflow }}",
           "blocks": [
             {
               "type": "section",
               "text": {
                 "type": "mrkdwn",
                 "text": "*Migration Status:* ${{ job.status }}\n*Branch:* ${{ github.ref_name }}\n*Environment:* ${{ github.ref == 'refs/heads/main' && 'Production' || 'Preview' }}\n*Migrations Applied:* ${{ steps.migrate.outputs.applied }}\n*Execution Time:* ${{ steps.migrate.outputs.time }}ms"
               }
             },
             {
               "type": "actions",
               "elements": [
                 {
                   "type": "button",
                   "text": {
                     "type": "plain_text",
                     "text": "View Logs"
                   },
                   "url": "${{ github.server_url }}/${{ github.repository }}/actions/runs/${{ github.run_id }}"
                 }
               ]
             }
           ]
         }
     env:
       SLACK_WEBHOOK_URL: ${{ secrets.SLACK_WEBHOOK_URL }}
   ```

3. **Email Alert with Details**:
   ```yaml
   - name: Send email alert
     if: failure()
     uses: dawidd6/action-send-mail@v3
     with:
       server_address: smtp.gmail.com
       server_port: 465
       username: ${{ secrets.EMAIL_USERNAME }}
       password: ${{ secrets.EMAIL_PASSWORD }}
       subject: "🚨 Migration Failed: ${{ github.ref_name }}"
       to: team@example.com
       from: github-actions@example.com
       body: |
         Migration failed on branch ${{ github.ref_name }}
         
         Commit: ${{ github.sha }}
         Environment: ${{ github.ref == 'refs/heads/main' && 'Production' || 'Preview' }}
         
         View logs: ${{ github.server_url }}/${{ github.repository }}/actions/runs/${{ github.run_id }}
         
         Please investigate immediately.
   ```

### Monitoring Service Integration

1. **Datadog Integration**:
   ```yaml
   - name: Send metrics to Datadog
     if: always()
     run: |
       curl -X POST "https://api.datadoghq.com/api/v1/series" \
         -H "Content-Type: application/json" \
         -H "DD-API-KEY: ${{ secrets.DATADOG_API_KEY }}" \
         -d '{
           "series": [
             {
               "metric": "migration.execution_time",
               "points": [['$(date +%s)', ${{ steps.migrate.outputs.time }}]],
               "type": "gauge",
               "tags": ["env:${{ github.ref == 'refs/heads/main' && 'production' || 'preview' }}", "branch:${{ github.ref_name }}"]
             },
             {
               "metric": "migration.success",
               "points": [['$(date +%s)', ${{ job.status == 'success' && 1 || 0 }}]],
               "type": "gauge",
               "tags": ["env:${{ github.ref == 'refs/heads/main' && 'production' || 'preview' }}", "branch:${{ github.ref_name }}"]
             }
           ]
         }'
   ```

2. **Prometheus Pushgateway**:
   ```yaml
   - name: Push metrics to Prometheus
     if: always()
     run: |
       cat <<EOF | curl --data-binary @- http://pushgateway:9091/metrics/job/migrations
       # TYPE migration_execution_time gauge
       migration_execution_time{branch="${{ github.ref_name }}",env="${{ github.ref == 'refs/heads/main' && 'production' || 'preview' }}"} ${{ steps.migrate.outputs.time }}
       # TYPE migration_success gauge
       migration_success{branch="${{ github.ref_name }}",env="${{ github.ref == 'refs/heads/main' && 'production' || 'preview' }}"} ${{ job.status == 'success' && 1 || 0 }}
       EOF
   ```

3. **CloudWatch Metrics**:
   ```yaml
   - name: Send metrics to CloudWatch
     if: always()
     run: |
       aws cloudwatch put-metric-data \
         --namespace "Migrations" \
         --metric-name "ExecutionTime" \
         --value ${{ steps.migrate.outputs.time }} \
         --unit Milliseconds \
         --dimensions Branch=${{ github.ref_name }},Environment=${{ github.ref == 'refs/heads/main' && 'production' || 'preview' }}
       
       aws cloudwatch put-metric-data \
         --namespace "Migrations" \
         --metric-name "Success" \
         --value ${{ job.status == 'success' && 1 || 0 }} \
         --unit Count \
         --dimensions Branch=${{ github.ref_name }},Environment=${{ github.ref == 'refs/heads/main' && 'production' || 'preview' }}
     env:
       AWS_ACCESS_KEY_ID: ${{ secrets.AWS_ACCESS_KEY_ID }}
       AWS_SECRET_ACCESS_KEY: ${{ secrets.AWS_SECRET_ACCESS_KEY }}
       AWS_DEFAULT_REGION: us-east-1
   ```

## Migration Execution Metrics

### Metrics Overview

The migration script generates comprehensive metrics for monitoring and alerting:

1. **Execution Metrics**:
   - Total execution time (milliseconds)
   - Average time per migration
   - Success rate (percentage)
   - Migrations applied count
   - Migrations skipped count
   - Migrations failed count
   - Invalid/skipped files count

2. **Context Metadata**:
   - Timestamp (ISO 8601 format)
   - Environment (Production/Preview/Development)
   - Branch name
   - Commit SHA (short)

3. **Metrics Output Format**:
   ```json
   {
     "timestamp": "2024-01-15T10:30:00.000Z",
     "environment": "Production",
     "branch": "main",
     "commit": "abc1234",
     "success": true,
     "executionTimeMs": 1250,
     "migrationsApplied": 3,
     "migrationsSkipped": 5,
     "migrationsFailed": 0,
     "totalMigrationFiles": 8,
     "invalidFiles": 0,
     "averageTimePerMigrationMs": 417,
     "successRate": 100
   }
   ```

### Accessing Metrics

1. **GitHub Actions Logs**:
   - Navigate to Actions → Deploy workflow → Select run
   - Expand "Run database migrations" step
   - Scroll to bottom to find "📊 Metrics JSON" section
   - Copy JSON for analysis or monitoring tools

2. **Automated Parsing**:
   - Use GitHub Actions API to fetch workflow logs
   - Parse JSON metrics from log output
   - Store in monitoring database (e.g., Prometheus, CloudWatch)
   - Create dashboards and alerts based on metrics

3. **Example: Parsing Metrics with GitHub CLI**:
   ```bash
   # Get latest workflow run logs
   gh run view --log | grep "Metrics JSON" -A 15
   
   # Extract JSON metrics
   gh run view --log | sed -n '/Metrics JSON/,/^━/p' | grep -E '^\{' -A 15
   ```

### Metrics Analysis

1. **Performance Monitoring**:
   - Track `executionTimeMs` over time
   - Alert if execution time exceeds threshold (e.g., > 30 seconds)
   - Identify slow migrations for optimization

2. **Success Rate Tracking**:
   - Monitor `successRate` across deployments
   - Alert on failures (`success: false`)
   - Track failure patterns by branch/environment

3. **Migration Volume**:
   - Track `migrationsApplied` per deployment
   - Identify large batches that may need review
   - Monitor `invalidFiles` for quality issues

4. **Trend Analysis**:
   - Compare metrics across time periods
   - Identify performance degradation
   - Optimize migration strategies based on data

### Setting Up Metrics Collection

1. **Option 1: GitHub Actions Artifacts**:
   ```yaml
   - name: Save migration metrics
     if: always()
     run: |
       echo "$METRICS_JSON" > migration-metrics.json
     env:
       METRICS_JSON: ${{ steps.migrate.outputs.metrics }}
   
   - name: Upload metrics artifact
     if: always()
     uses: actions/upload-artifact@v4
     with:
       name: migration-metrics
       path: migration-metrics.json
       retention-days: 90
   ```

2. **Option 2: Send to Monitoring Service**:
   ```yaml
   - name: Send metrics to monitoring
     if: always()
     run: |
       curl -X POST https://your-monitoring-service.com/metrics \
         -H "Content-Type: application/json" \
         -H "Authorization: Bearer ${{ secrets.MONITORING_TOKEN }}" \
         -d "$METRICS_JSON"
     env:
       METRICS_JSON: ${{ steps.migrate.outputs.metrics }}
   ```

3. **Option 3: Store in Database**:
   ```yaml
   - name: Store metrics in database
     if: always()
     run: |
       bun run scripts/store-metrics.ts
     env:
       METRICS_JSON: ${{ steps.migrate.outputs.metrics }}
       DATABASE_URL: ${{ secrets.METRICS_DB_URL }}
   ```

## Quick Start: Setting Up Monitoring

### Step 1: Add Status Badges

2. Add badges at the top (replace `YOUR_USERNAME` and `YOUR_REPO`):

## Troubleshooting Monitoring Issues

### Status Badges Not Updating

**Symptoms**: Badges show outdated status or "unknown"

**Possible Causes**:
- Workflow file renamed or moved
- Badge URL incorrect
- GitHub Actions cache issue

**Resolution**:
1. Verify workflow file path matches badge URL
2. Check workflow has run recently
3. Clear browser cache and refresh
4. Regenerate badge URL from GitHub Actions page

### Metrics Not Appearing in Logs

**Symptoms**: Migration runs but no metrics JSON in output

**Possible Causes**:
- Migration script version outdated
- Log output truncated
- Script error before metrics generation

**Resolution**:
1. Verify migration script is latest version
2. Check for script errors earlier in logs
3. Review script execution completed successfully
4. Increase log retention if needed

### Alerts Not Triggering

**Symptoms**: Failures occur but no alerts received

**Possible Causes**:
- Webhook URL incorrect or expired
- Secrets not configured
- Alert condition not met
- Service outage

**Resolution**:
1. Verify webhook URL is correct and active
2. Check secrets are configured in GitHub
3. Review alert condition logic
4. Test webhook manually with curl
5. Check monitoring service status
