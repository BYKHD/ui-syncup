# UI SyncUp CLI – Self‑Hosting & Operations Guide

This document describes the **UI SyncUp CLI** designed to support self‑hosting, local development, and production operations while keeping configuration simple and safe.

---

## Goals of the CLI

- Make self‑hosting easy and predictable
- Reduce setup errors and GitHub support issues
- Support CI/CD and infrastructure automation
- Avoid replacing the Web Admin UI

**Principle:**
> CLI handles lifecycle & infrastructure.
> Web UI handles configuration & integrations.

---

## Installation

```bash
git clone https://github.com/your-org/ui-syncup.git
cd ui-syncup
bun install
```

Run via Bunx:

```bash
bunx ui-syncup <command>
```

## `ui-syncup init`

### Purpose
Initializes the project for self‑hosting with **safe defaults**.

### Usage
```bash
ui-syncup init
ui-syncup init --verbose
ui-syncup init --mode production    # Non-interactive production setup
```

### Options
| Option | Description |
|--------|-------------|
| `--mode <local\|production>` | Skip mode prompt; select setup mode directly |
| `--verbose` | Enable debug output |

### Flow
- Detects system requirements (Bun, Docker, Supabase CLI, ports)
- Prompts for setup mode (or uses `--mode` flag):
  - **Local** — recommended for development
  - **Production** — for external/self-hosted services
- Generates required files and folders

### Files Created
```
.env.local (or .env.production)
docker-compose.override.yml
/storage/uploads
/storage/avatars
ui-syncup.config.json
```

### Example Default Config (Local)
```env
DATABASE_URL=postgresql://postgres:postgres@127.0.0.1:54322/postgres
SUPABASE_URL=http://127.0.0.1:54321
BETTER_AUTH_SECRET=<auto-generated>
STORAGE_ENDPOINT=http://127.0.0.1:9000
```

No external credentials required at this stage.

### CI/IaC Usage

For automated production bootstrap without prompts:

```bash
CI=1 bun ./cli/index.ts init --mode production
```

This creates a production scaffold (`.env.production`) that must be populated with real values before deployment.

## `ui-syncup up`

### Purpose
Starts the local development stack.

### Usage
```bash
ui-syncup up
ui-syncup up --verbose
```

### What it does
1. Starts Supabase services (database, auth, realtime, etc.)
2. Waits for database readiness
3. Starts MinIO storage service (when `docker-compose.minio.yml` exists)
4. Waits for MinIO health check (`/minio/health/live`)
5. Displays service URLs and readiness status

### Output
```text
✔ Supabase started successfully

Services ready:
  Supabase Studio:  http://127.0.0.1:54323
  API URL:          http://127.0.0.1:54321
  Database:         postgresql://postgres:postgres@127.0.0.1:54322/postgres
  MinIO Console:    http://127.0.0.1:9001

🚀 Local stack ready (core + storage)
```

> **Note:** If MinIO fails to start, the CLI warns but does not block the rest of the stack. Core services will still be available.

## `ui-syncup down`

### Purpose
Stops all local services.

### Usage
```bash
ui-syncup down
ui-syncup down --verbose
```

### What it does
- Stops MinIO storage service (if running)
- Stops Supabase services via `supabase stop`

---

## `ui-syncup reset`

### Purpose
Safely wipe **local** data while preserving configuration.

### Usage
```bash
ui-syncup reset
```

### Safety
- Blocked in production environments (inspects `.env` files for production indicators)
- Requires manual confirmation
- No `--force` option

### What it resets
| Resource | Action |
|----------|--------|
| MinIO storage service | Stopped |
| Storage directories | Cleared |
| Docker containers | Removed |
| Supabase services | Stopped |

### What it preserves
- `.env.local` / `.env.production`
- `docker-compose.override.yml`
- `ui-syncup.config.json`

---

## `ui-syncup purge`

### Purpose
Aggressive factory reset for **development only**. Removes everything including configuration to start from absolute zero.

### Usage
```bash
ui-syncup purge
```

### Safety
- Blocked in production environments
- Requires manual confirmation with typed confirmation phrase
- Warns about irreversible action

### What it removes (everything)
| Resource | Action |
|----------|--------|
| Local database | Wiped |
| Uploaded files (`/storage/*`) | Removed |
| MinIO containers, volumes, images | Purged |
| Docker containers and volumes | Removed |
| Configuration files (`.env*`, `docker-compose.override.yml`) | Removed |
| Generated directories | Removed |

### Comparison: `reset` vs `purge`
| Aspect | `reset` | `purge` |
|--------|---------|--------|
| Database | ✓ Wiped | ✓ Wiped |
| Uploaded files | ✓ Wiped | ✓ Wiped |
| Docker containers | ✓ Removed | ✓ Removed |
| Docker volumes | ✗ Kept | ✓ Removed |
| Docker images | ✗ Kept | ✓ Removed |
| `.env*` files | ✗ Kept | ✓ Removed |
| `docker-compose.override.yml` | ✗ Kept | ✓ Removed |
| Use case | Fresh data | Undo `init` |

## Updating UI SyncUp

UI SyncUp uses **Git-based updates** instead of a dedicated `update` command.

### Update Workflow

```bash
# 1. Stop services
ui-syncup down

# 2. Pull latest version
git fetch origin
git checkout v2.1.0    # or: git pull origin main

# 3. Update dependencies
bun install

# 4. Restart services
ui-syncup up
```

## Configuration Ownership Model

| Config Type | Location |
|-------------|----------|
| Secrets | `.env*` |
| Provider choice | Database |
| Feature flags | Database |
| Infrastructure | Environment |

CLI **writes config**, Web UI **manages runtime settings**.

---

## What the CLI Does NOT Do

- ❌ No business logic
- ❌ No user or project management
- ❌ No daily admin workflows
- ❌ No provider lock‑in

---

## Design Philosophy

- Local‑first defaults
- External SaaS is opt‑in
- Predictable commands
- Safe by default
- Web UI remains primary control plane

## Summary

UI SyncUp CLI is intentionally **minimal**, focused on:
- Setup (`init`)
- Lifecycle (`up`, `down`)
- Recovery (`reset`, `purge`)
- Automation (`--mode`, `CI=1`)

This keeps the product approachable while still supporting serious self‑hosting and production use cases.
