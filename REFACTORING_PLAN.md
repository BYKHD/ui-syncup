# Refactoring Plan: Installation & Distribution

## Context

UI SyncUp is a self-hosted open-source app (like Plausible, Umami, Coolify).
Users deploy the **whole app** with their own config (Supabase, S3, email).
They never modify source code, and they want to receive updates easily.

The current architecture has one root problem:

> **The CLI lives inside the app repo.** Users must clone the full source code just to run `init` or `up`. The `findProjectRoot()` check in every command enforces this.

---

## Target Architecture

```
User's server/machine          What you publish
─────────────────────          ──────────────────────────────────
my-ui-syncup/                  npm: ui-syncup  (CLI only)
  .env                 ←───    ghcr.io/bykhd/ui-syncup:v0.x.x  (Docker image)
  docker-compose.yml   ←───
```

**User flow after refactor:**

```bash
# Install (one-time, from any empty directory)
bunx ui-syncup init

# Start
bunx ui-syncup up

# Update (get new version)
bunx ui-syncup update
```

No `install.sh`. No git clone. No source code on the user's machine.

---

## What Needs To Change

### 1. Publish CLI to npm separately [DONE]

**Problem:** `package.json` is `"private": true` — the CLI is never published.

**What to do:**
- Create `cli/package.json` as a standalone package (`name: "ui-syncup"`, `"private": false`)
- The root `package.json` stays `"private": true` (it is the app, not a package)
- Add a GitHub Actions workflow: on release tag → `npm publish` from `cli/`

**Files affected:**
- `cli/package.json` (new)
- `.github/workflows/` (new publish workflow)

---

### 2. Remove `findProjectRoot()` from user-facing commands

**Problem:** Every command (`init`, `up`, `down`, `reset`, `purge`) calls `findProjectRoot()` which checks for `package.json` with `name: "ui-syncup"`. This means the CLI only works inside the cloned source repo.

**What to do:**
- Change detection logic: look for `ui-syncup.config.json` instead (generated by `init`)
- `init` is the only command that runs without `ui-syncup.config.json` — it creates it
- `up`, `down`, `reset`, `purge` look for `ui-syncup.config.json` in cwd

**Files affected:**
- `cli/lib/config.ts` (change `findProjectRoot` logic)
- `cli/commands/up.ts`, `down.ts`, `reset.ts`, `purge.ts` (update check)

---

### 3. Build & publish Docker image on release

**Problem:** No Docker image exists. `init --mode production` generates a `Dockerfile` for users to build themselves — this means every user builds the image from source, which is slow, error-prone, and means they have source code.

**What to do:**
- Add GitHub Actions workflow: on release tag → build Docker image → push to `ghcr.io/bykhd/ui-syncup:VERSION` and `:latest`
- `init` generates a `docker-compose.yml` that pulls the pre-built image (no build step)

**Files affected:**
- `.github/workflows/docker-release.yml` (new)
- `cli/templates/` (new `docker-compose.template.yml` referencing the image)
- `cli/commands/init.ts` (generate `docker-compose.yml` instead of `Dockerfile`)

---

### 4. Add `update` command to CLI

**Problem:** There is no `update` command. Users have no way to get new versions without re-cloning or running `install.sh --upgrade`.

**What to do:**
- Add `bunx ui-syncup update` command
- It does: `docker compose pull && docker compose up -d`
- Optionally accepts a version: `bunx ui-syncup update v0.3.0`

**Files affected:**
- `cli/commands/update.ts` (new)
- `cli/commands/index.ts` (register the command)

---

### 5. Simplify or remove `install.sh`

**Problem:** `install.sh` exists to bootstrap the install, but after this refactor it is replaced by `bunx ui-syncup init`.

**Options:**
- A. Delete `install.sh` entirely. Update README to `bunx ui-syncup init`.
- B. Keep a minimal `install.sh` that only checks for bun and then runs `bunx ui-syncup init` (useful for CI/automation).

---

### 6. Separate local dev mode from self-host mode

**Current state:** `up` starts Supabase CLI + MinIO locally — this is for contributors developing ui-syncup itself, not for self-hosters.

**After refactor, two clear modes:**

| Command | Who uses it | What it does |
|---|---|---|
| `bunx ui-syncup up` | Self-hosters | `docker compose up` (pulls image) |
| `bun run dev` (from cloned repo) | Contributors | Supabase CLI + MinIO + Next.js dev |

The `up` command should change behavior based on `ui-syncup.config.json` mode:
- `mode: local` (contributor) → current behavior (Supabase CLI)
- `mode: production` (self-hoster) → `docker compose up`

---

## Decisions

| # | Question | Decision | Rationale |
|---|---|---|---|
| Q1 | CLI package name | `ui-syncup` (unscoped) | Matches brand; `bunx ui-syncup init` reads cleanly; publish immediately to avoid squatting |
| Q2 | Docker registry | `ghcr.io/bykhd/ui-syncup` | Free, no rate limits, native GitHub Actions auth, co-located with releases |
| Q3 | Supabase for self-hosters | External Supabase (cloud or self-hosted) | Supabase-in-Docker is a second product; `init` wizard collects Supabase URL + keys |
| Q4 | DB migrations | Auto-run on container startup via `entrypoint.sh` | Non-technical users must not need a separate step; precedent: Coolify, Umami, Plausible |
| Q5 | Backward compatibility | Clean break (new major version) | Old install path (`install.sh`) is invalid after refactor anyway; provide migration guide in README |

---

## Suggested Order of Work

1. Answer open questions above
2. Create `cli/package.json` and test publishing flow (dry-run)
3. Add Docker image build workflow (`.github/workflows/docker-release.yml`)
4. Refactor `findProjectRoot()` → `findConfigFile()`
5. Update `init` to generate `docker-compose.yml` (image-based, not build-based)
6. Add `update` command
7. Update `up` to handle both local and production modes
8. Simplify / replace `install.sh`
9. Update README and docs

---

## What Does NOT Change

- The app source code (Next.js, Drizzle, auth, etc.)
- Local development flow for contributors (`bun dev`, Supabase CLI)
- The `init` wizard UX (questions, env file generation)
- CI workflow (`ci.yml`)
- The `down`, `reset`, `purge` commands (logic stays the same, only the root detection changes)
